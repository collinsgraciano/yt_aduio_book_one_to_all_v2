# 项目迁移指南

## 1. 概述

本文档指导你将有声书 YouTube 频道管理系统从一台 VPS（源 VPS）完整迁移到另一台 VPS（目标 VPS）。

### 迁移内容

| 内容 | 说明 | 必须 |
|------|------|------|
| PostgreSQL 数据库 | 所有表结构和数据（books、channels、configs 等） | ✅ |
| .env 配置文件 | `SECRET_KEY`、`POSTGRES_PASSWORD`、`BASE_URL` 等 | ✅ |
| relay-data 卷 | HF 中继配置（`relay_config.json`） | ✅ |
| output_data 卷 | 生成的音频输出文件 | 可选（可能很大） |

### 关键注意事项

- **`SECRET_KEY` 必须一致**：数据库中 `channels.oauth_client_secret` 等敏感数据使用 `SECRET_KEY` 加密，迁移后必须使用相同的 `SECRET_KEY`，否则加密数据无法解密。
- **`POSTGRES_PASSWORD` 必须一致**：通过复制 `.env` 文件自动保证。
- **`BASE_URL` 需要更新**：VPS IP 变更后，需修改 `.env` 中的 `BASE_URL` 并更新 Google Cloud Console 中的 OAuth 重定向 URI。
- **YouTube OAuth Token 可继续使用**：`youtube_credentials` 表中的 `refresh_token` 不依赖回调地址，迁移后仍可正常刷新。
- **新 OAuth 授权需更新回调地址**：如需重新授权频道，必须先更新 Google Cloud Console 中的重定向 URI。

## 2. 前置条件

### 源 VPS

- 项目正常运行
- Docker + Docker Compose 已安装
- PostgreSQL 容器（`audiobook_postgres`）正在运行

### 目标 VPS

- Docker + Docker Compose 已安装
- 已克隆项目仓库（`git clone`）
- 有足够的磁盘空间（至少 5GB，含输出文件则更多）
- 端口 59386（Web）、38080（中继）、5432（PostgreSQL）未被占用

## 3. 迁移步骤

### 步骤 1：在源 VPS 上导出

SSH 登录源 VPS，进入项目目录：

```bash
cd /root/audiobook

# 导出（不含音频输出文件，推荐）
bash scripts/migrate_export.sh

# 或导出（含音频输出文件，文件可能很大）
bash scripts/migrate_export.sh --with-output
```

脚本执行流程：

```
[1/5] 数据库备份        → db_backup.sql.gz
[2/5] 导出 .env 配置    → env_backup
[3/5] 导出 relay-data   → relay_data.tar.gz
[4/5] 导出 output_data  → output_data.tar.gz（仅 --with-output）
[5/5] 创建迁移包        → migration_bundle_YYYYMMDD_HHMMSS.tar.gz
```

导出完成后，会在项目目录下生成迁移包文件：

```
migration_bundle_20260721_120000.tar.gz
```

### 步骤 2：传输迁移包到目标 VPS

**方法 1：从源 VPS 推送**

```bash
scp /root/audiobook/migration_bundle_YYYYMMDD_HHMMSS.tar.gz root@TARGET_VPS_IP:/root/audiobook/
```

**方法 2：在目标 VPS 上拉取**

```bash
scp root@SOURCE_VPS_IP:/root/audiobook/migration_bundle_YYYYMMDD_HHMMSS.tar.gz /root/audiobook/
```

> 如果目标 VPS 尚未克隆仓库，先执行：
> ```bash
> git clone https://github.com/YOUR_USER/YOUR_REPO.git /root/audiobook
> ```
> 然后将迁移包放到 `/root/audiobook/` 目录下。

### 步骤 3：在目标 VPS 上导入

SSH 登录目标 VPS，进入项目目录：

```bash
cd /root/audiobook

# 确保代码是最新的
git pull

# 执行导入（会提示确认）
bash scripts/migrate_import.sh migration_bundle_YYYYMMDD_HHMMSS.tar.gz

# 或跳过确认提示（自动化场景）
bash scripts/migrate_import.sh migration_bundle_YYYYMMDD_HHMMSS.tar.gz --force
```

脚本执行流程：

```
[1/9] 解包迁移包
[2/9] 恢复 .env 配置（自动备份现有 .env）
[3/9] 清理旧环境（docker-compose down -v）
[4/9] 构建并启动服务（首次约 5-10 分钟）
[5/9] 等待 PostgreSQL 就绪
[6/9] 恢复数据库（DROP + CREATE + 导入）
[7/9] 恢复 relay-data 卷
[8/9] 恢复 output_data 卷（如包含）
[9/9] 启动服务并验证
```

导入完成后，脚本会显示数据库表行数对比和服务状态。

## 4. 迁移后配置

### 4.1 修改 BASE_URL

VPS IP 变更后，必须更新 `BASE_URL`：

```bash
cd /root/audiobook
nano .env
```

修改这一行：

```env
BASE_URL=http://NEW_VPS_IP:59386
```

### 4.2 更新 Google Cloud Console OAuth 重定向 URI

1. 访问 [Google Cloud Console](https://console.cloud.google.com/apis/credentials)
2. 进入 **APIs & Services** → **Credentials**
3. 编辑你的 OAuth 2.0 Client ID
4. 在「授权重定向 URI」中添加：

```
http://NEW_VPS_IP:59386/api/oauth/callback
```

5. 保存

> 旧的 URI 可以保留不删，不影响使用。

### 4.3 重启服务

```bash
cd /root/audiobook
docker-compose restart web
```

### 4.4 停止源 VPS 的服务（可选但推荐）

为避免两台 VPS 同时运行导致冲突（如同时连接 HF Worker），建议停止源 VPS 的服务：

```bash
# 在源 VPS 上执行
cd /root/audiobook
docker-compose down
```

## 5. 验证清单

迁移完成后，逐项检查：

- [ ] **Web 界面可访问**：浏览器打开 `http://NEW_VPS_IP:59386`
- [ ] **HF 中继面板可访问**：浏览器打开 `http://NEW_VPS_IP:38080`
- [ ] **数据库表行数一致**：导入脚本输出的表行数与源 VPS 一致
- [ ] **YouTube 频道 OAuth 状态正常**：在 Web 界面「频道管理」中检查授权状态
- [ ] **全局设置完整**：在 Web 界面「设置」中检查 `TG_BOT_TOKEN`、`MODELSCOPE_TOKEN` 等
- [ ] **HF 中继 Worker 配置**：检查中继面板的 Worker URL 配置是否正确
- [ ] **测试运行任务**：选择一本书，运行完整流水线，确认各环节正常

### 验证命令

```bash
# 查看服务状态
docker-compose ps

# 查看数据库表行数
docker exec audiobook_postgres psql -U audiobook_app -d audiobook -c "
    SELECT relname AS table_name, n_live_tup AS row_count
    FROM pg_stat_user_tables
    ORDER BY n_live_tup DESC;
"

# 检查 YouTube 凭证
docker exec audiobook_postgres psql -U audiobook_app -d audiobook -c "
    SELECT channel_name, updated_at FROM youtube_credentials;
"

# 检查频道注册
docker exec audiobook_postgres psql -U audiobook_app -d audiobook -c "
    SELECT channel_name, display_name, oauth_status FROM channels;
"

# 检查全局设置
docker exec audiobook_postgres psql -U audiobook_app -d audiobook -c "
    SELECT setting_key, CASE WHEN is_secret THEN '***' ELSE setting_value END AS value
    FROM global_settings ORDER BY setting_key;
"

# 查看 Web 服务日志
docker-compose logs --tail 30 web

# 查看中继服务日志
docker-compose logs --tail 30 vps-relay
```

## 6. 常见问题

### Q: 迁移后 Web 服务无法连接数据库？

**A**：检查 `.env` 中的 `POSTGRES_PASSWORD` 是否与源 VPS 一致。导入脚本会自动复制 `.env`，但如果目标 VPS 之前有残留的 PostgreSQL 数据卷，可能导致密码不匹配。

解决方法：
```bash
docker-compose down -v
docker-compose up -d --build
# 然后重新运行导入脚本
bash scripts/migrate_import.sh migration_bundle_YYYYMMDD_HHMMSS.tar.gz --force
```

### Q: 迁移后 YouTube OAuth 失效？

**A**：OAuth `refresh_token` 存储在 `youtube_credentials` 表中，不依赖回调地址，迁移后应该可以正常刷新。

如果 Token 已过期且无法刷新：
1. 确认 `SECRET_KEY` 与源 VPS 一致（检查 `.env`）
2. 在 Google Cloud Console 更新重定向 URI
3. 在 Web 界面重新授权频道

### Q: 迁移后加密数据（OAuth client_secret）无法读取？

**A**：`SECRET_KEY` 用于加密 `channels.oauth_client_secret`。确保 `.env` 中的 `SECRET_KEY` 与源 VPS 完全一致：

```bash
# 对比源 VPS 和目标 VPS 的 SECRET_KEY
grep SECRET_KEY .env
```

如果不一致，说明 `.env` 未正确恢复。重新从迁移包恢复：
```bash
tar xzf migration_bundle_YYYYMMDD_HHMMSS.tar.gz -C /tmp/migration
cp /tmp/migration/env_backup .env
docker-compose restart web
```

### Q: 音频输出文件没有迁移？

**A**：默认不迁移音频输出文件（可能很大）。如需迁移，在源 VPS 导出时使用 `--with-output` 参数：

```bash
bash scripts/migrate_export.sh --with-output
```

> 注意：输出文件可能占用大量磁盘空间和传输带宽。

### Q: 迁移包太大，SCP 传输很慢/中断？

**A**：可以使用 `rsync` 替代 `scp`（支持断点续传）：

```bash
rsync -avP --partial /root/audiobook/migration_bundle_YYYYMMDD_HHMMSS.tar.gz root@TARGET_VPS_IP:/root/audiobook/
```

或先压缩再传输（如果迁移包包含 `--with-output` 的音频文件）：

```bash
# 在源 VPS 上分卷压缩
split -b 500M migration_bundle_YYYYMMDD_HHMMSS.tar.gz migration_bundle_part_

# 传输各分卷
scp migration_bundle_part_* root@TARGET_VPS_IP:/root/audiobook/

# 在目标 VPS 上合并
cat migration_bundle_part_* > migration_bundle_YYYYMMDD_HHMMSS.tar.gz
```

### Q: 如何查看迁移包内容？

```bash
tar tzf migration_bundle_YYYYMMDD_HHMMSS.tar.gz
```

### Q: 目标 VPS 端口被占用？

```bash
# 检查端口占用
ss -tlnp | grep -E '59386|38080|5432'

# 如需修改端口，编辑 docker-compose.yml 中的 ports 映射
```

### Q: 目标 VPS 架构不同（ARM vs x86）？

DeepFilter 降噪二进制仅支持 `x86_64`。如果目标 VPS 是 ARM 架构（如 Ampere、Graviton），降噪功能将不可用，但其他功能正常。

## 7. 回滚方案

### 场景 1：迁移失败，目标 VPS 需恢复到迁移前状态

```bash
cd /root/audiobook

# 恢复 .env 备份（导入脚本会自动备份）
cp .env.bak_YYYYMMDD_HHMMSS .env

# 重启服务
docker-compose restart
```

### 场景 2：目标 VPS 需完全重置

```bash
cd /root/audiobook

# 停止并删除所有容器和数据卷
docker-compose down -v

# 重新部署
bash scripts/git-server-deploy.sh
```

### 场景 3：回退到源 VPS

如果迁移后发现问题，需要回退到源 VPS 运行：

```bash
# 在源 VPS 上重新启动服务
cd /root/audiobook
docker-compose up -d
```

> 前提：源 VPS 的数据未被修改（迁移导出不会删除源 VPS 数据）。

## 8. 迁移包结构说明

```
migration_bundle_YYYYMMDD_HHMMSS.tar.gz
├── db_backup.sql.gz        # 数据库完整备份（表结构 + 数据）
├── env_backup              # .env 配置文件
├── relay_data.tar.gz       # relay-data 卷内容（HF 中继配置）
├── output_data.tar.gz      # output_data 卷内容（可选，音频文件）
└── manifest.txt            # 迁移清单（时间戳、主机名、表行数）
```

`manifest.txt` 示例：

```
MIGRATION_VERSION=1.0
EXPORT_TIMESTAMP=20260721_120000
SOURCE_HOSTNAME=vps-a
INCLUDE_OUTPUT_DATA=false

TABLE_COUNTS_BEGIN
audiobook_chapters = 5678
books = 1234
channel_configs = 3
channel_runtime_settings = 150
channels = 3
global_settings = 7
hf_jobs = 0
hf_worker_stats = 2
modelscope_tokens = 2
oauth_states = 0
run_task_logs = 89234
run_tasks = 56
task_queue = 12
youtube_credentials = 3
TABLE_COUNTS_END
```
